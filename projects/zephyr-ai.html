A dual-interface personal AI assistant.

Personal AI Productivity Multi-LLM

Press Ctrl+Alt+Z for an instant popup. Launch the Electron dashboard for the full workspace. Both surfaces talk to one Flask API and one SQLite memory store, so the assistant feels like one tool no matter where it is invoked from.

1. Abstract

Personal AI assistants usually force a tradeoff. A lightweight popup is fast but cannot do much. A heavy desktop app can do everything but takes too long to open for the small task that triggered the impulse to ask. Zephyr ships both, sharing one backend so memory and integrations stay consistent across surfaces. Quick command, popup. Complex management, dashboard. The user never feels like the assistant has two minds.

The implementation is a three-tier architecture. A Tkinter popup on a system-wide hotkey (Ctrl+Alt+Z) for instant capture. A Flask API on localhost that owns all state. An Electron + React dashboard for the full workspace: conversation history, memory browser, project board, settings. The popup and the dashboard never talk to each other; both speak only to the API. SQLite holds persistent memory, integrations sit behind their own endpoints, and a multi-LLM router fronts Groq Llama 3.1 8B, OpenAI, and Gemini so a single provider outage does not take the assistant down.

The novelty is the surface design, not the LLM. Multi-LLM routing exists, SQLite-backed memory exists, system-tray apps exist. The contribution is the discipline that keeps two interfaces backed by one backend feel like a single product: the API is the only abstraction boundary, every integration is a single Python module that registers two functions, and the dashboard treats popup activity as just another client of the same endpoints.

  • Three-tier architecture. Tkinter popup, Flask API on localhost:5000, Electron + React dashboard. One backend, two interfaces, zero direct UI-to-UI traffic.
  • Sub-100 ms memory lookups from SQLite across thousands of stored facts, with relevance scoring that prioritises recent and frequently-accessed entries.
  • Multi-LLM router with provider fallback. Groq Llama 3.1 8B is the default for speed. OpenAI and Gemini back it up automatically when the primary is unavailable.
  • Streaming responses to both surfaces via the same Flask endpoint. Perceived latency is dominated by network, not by the architecture.
  • Plugin-style integrations. A new handler is one Python module with a can_handle() and an execute(). No core changes, no router edits.
  • Spotify, Google Calendar, WeatherAPI, and local file search as first-party integrations, all behind their own routed endpoints.

2. Introduction

Voice assistants on phones are fast but shallow. Desktop AI apps are deep but slow to launch and slow to navigate. The user's tasks do not fit either shape cleanly; they span the same memory, but they want different interfaces depending on what they are doing.

  • Quick tasks need a fast capture surface. "Remind me to email the registrar tomorrow." The cognitive cost of opening a dashboard, navigating to the right pane, and clicking through forms kills the impulse before the thought reaches the keyboard.
  • Complex tasks need a workspace. "Pull up everything we logged about the rehab study, group by week, export to markdown." A popup is the wrong shape for that kind of exploration; a dashboard is the right one.
  • Both tasks share memory. The assistant cannot have two personas. What you tell the popup must be available to the dashboard, and vice versa, without manual sync.

2.1 Why three tiers, not two

The naive design is one process that ships both UIs. That fails as soon as either UI wants to update independently, run on a different machine, or be replaced by a third client (a mobile companion, a Slack bot, a voice device). The three-tier split puts the API in the middle as the only stable contract, and each UI becomes a thin client of that contract. Cost: an extra process. Benefit: the assistant can grow without rebuilding state management each time.

  • The API is the abstraction boundary. All state lives in the Flask server. The popup is a UI, not a backend with a UI on top of it.
  • The popup and dashboard never share code beyond the API contract. Tkinter and Electron speak the same JSON over HTTP, and that is all they have in common.
  • New surfaces are cheap. A mobile companion or a Slack endpoint adds another client of the same API rather than another state owner.

2.2 What is actually new

Tkinter is decades old. Flask is decades old. Electron is a known quantity. The LLM providers are off the shelf. The contribution is the structural design that holds those pieces together into something that feels like one product.

  • Two interfaces, one backend, zero state drift. Memory written by the popup is visible to the dashboard within a single API roundtrip. There is no sync layer because there is no second source of truth.
  • Provider-agnostic LLM layer. The router treats Groq, OpenAI, and Gemini as interchangeable for the same task. Falling back across providers does not require the calling code to know it happened.
  • Plugin-shaped integrations. Every external service (Spotify, Calendar, Weather, file search) is wrapped as a handler that implements two functions. Adding a new one is a single file plus a single import.

Every section below is the engineering that makes those three commitments hold across two UI surfaces.

3. System

3.1 Tiny Zephyr (the popup)

  • Python + Tkinter with a dark theme, slide-in animation, and centring on the active monitor.
  • Global hotkey (Ctrl+Alt+Z) registered system-wide on Windows. Brings the window to focus from any state without stealing window order more than necessary.
  • Auto-hide on focus loss and a system-tray presence that keeps the hotkey reachable across sessions.
  • Voice input via microphone routed into the same NLP pipeline as typed input. The popup never branches on input modality past the capture step.
  • Streaming responses with concurrent text-to-speech. The assistant talks while the text materialises so the perceived latency stays under what the LLM's full generation takes.

3.2 Flask API bridge

Runs on localhost:5000, auto-started with Tiny Zephyr but addressable independently. Both UI surfaces speak only to the API; all state lives in this process.

  • POST /command is the natural-language entry point. Routes the input to the right handler, executes, and returns a structured response with both the answer and any side effects (memory writes, integration calls).
  • GET / POST / DELETE /memory provides full CRUD over the SQLite memory store, surfaced to both UIs identically.
  • GET / POST /projects handles project and idea tracking with tags and status. The dashboard's Kanban board reads from this endpoint; the popup writes to it by phrase ("add project: rewrite the portfolio").
  • Per-integration endpoints for Spotify control, Weather queries, and Calendar events. Each one is a thin wrapper over the upstream API with the assistant's response shaping bolted on.
  • WebSocket channel pushes live dashboard updates from popup activity, so a fact captured through the popup appears in the dashboard's memory browser without a refresh.

3.3 Electron dashboard

  • React 18 + Vite + TailwindCSS + Framer Motion, wrapped in Electron for native desktop integration (tray, autostart, OS notifications).
  • Conversation history with threaded view, search, filter, and JSON or Markdown export. The same conversation captured through the popup is rendered here without translation.
  • Memory browser with categorised facts, a tag cloud, manual edit affordances, and bulk cleanup for stale entries.
  • Project board as a Kanban surface with drag-and-drop reordering, Markdown descriptions, and attachments.
  • Settings panel for LLM API keys, integration credentials, theme toggles, and hotkey behaviour. All of this writes through the API rather than to local files.

3.4 Memory and routing

  • SQLite-backed memory holds learned facts, user preferences, conversation context, and project ideas in a single store.
  • Relevance scoring on retrieval prioritises recent and frequently-accessed memories, so a months-old captured fact does not crowd out something the user mentioned this morning.
  • Command router uses a three-tier escalation: exact-match on registered intents, fuzzy match on phrasing, then LLM intent classification when neither lands. The cheap tiers absorb most traffic.
  • Multi-LLM fallback. Groq Llama 3.1 8B is the default for latency. OpenAI and Gemini back it up. Provider failure triggers automatic re-routing with no user-visible interruption.

3.5 Integrations

  • Spotify via Spotipy. Play, pause, skip, "what is playing," queue from natural language.
  • WeatherAPI. System-location or explicit-city queries with contextual advice ("bring a jacket").
  • Google Calendar. Event listing and creation with natural-language date parsing ("add a study block tomorrow at 7 pm").
  • Local file search. "Find my resume," "open project folder" against a configured set of paths, so the assistant can act on local state, not just remote services.

4. Evaluation

Evaluation is functional and personal-use-driven, not benchmark-based. Zephyr is a tool the author uses daily, not a system aimed at a leaderboard; the numbers below are measured on the running install rather than against an external suite.

  • Memory lookups under 100 ms across thousands of stored facts, validated by direct timing on the API endpoint.
  • Multi-LLM fallback survives provider outages. When Groq has been unavailable, the router has rolled over to OpenAI without user-visible interruption.
  • Streaming responses materialise before the LLM has finished generating. The perceived latency floor is the network roundtrip to the provider, not the architecture.
  • Modular handler system. Each new integration is one Python module with two functions. Adding Spotify, Calendar, and Weather did not require core router changes.
  • Two surfaces, one memory. A fact captured through the popup is visible in the dashboard's memory browser within the next render, validated by direct observation during use.

The numbers above are measurements of this build's behaviour on its own protocol, not comparisons against another assistant. Personal-AI assistants do not have a settled benchmark suite the way ML models do, and Zephyr's design point (two interfaces, one backend, persistent memory) is not directly comparable to single-surface assistants like the OS-native ones. Read the results as evidence that the three-tier architecture works as a daily-use tool, not as a leaderboard claim.

5. Conclusion and Implications

The right interface for an AI assistant depends on the task, not the assistant. By splitting popup and dashboard but unifying state behind an API, Zephyr lets the user pick the interaction mode without giving up coherence. The downstream consequence is that any future surface (mobile companion, voice device, Slack bot, terminal client) becomes another client of the same backend rather than a fork of the assistant. The architecture also positions the build to absorb local LLMs (Ollama, llama.cpp, LM Studio) without redesign; the API becomes the abstraction boundary, and the provider behind it is interchangeable. For a personal tool that has to live in the user's daily workflow without becoming yet another app to manage, that boundary is the load-bearing one.

5.1 Limitations

  • Windows-only at present. The hotkey, tray, and OS-notification paths use Windows-specific APIs. macOS and Linux are reachable but unbuilt.
  • Single instance only. Two copies of the popup conflict on the global hotkey and on port 5000. There is no multi-user mode.
  • Cloud LLM dependence for the deep features. Offline mode covers file operations and cached memory; reasoning-heavy queries still require an upstream provider.
  • No end-to-end encryption on memory at rest. SQLite sits on local disk in plaintext; that is acceptable for a single-user local tool, not for a multi-device sync scenario.
  • Integrations cover what the author uses. Spotify, Calendar, Weather, and file search are first-party, but a user with a different stack would need to write their own handler module.

5.2 Future Work

  • macOS and Linux ports. pynput for hotkeys, AppKit and GTK for the platform-specific shims, identical Flask backend.
  • Local LLM support via Ollama or llama.cpp for fully offline operation. The multi-LLM router already abstracts the provider, so this is a new backend, not a redesign.
  • Optional encrypted cloud sync with a React Native mobile companion as the first additional surface.
  • Plugin marketplace with sandboxed third-party handlers, so users can extend the integration set without forking the core.
  • Voice-only mode on top of the existing speech-to-text path, for a hands-free workflow.
