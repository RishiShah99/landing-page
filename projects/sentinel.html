Hardware-aware static analysis for embedded code.

Developer Tools Embedded Systems Language Server

A VS Code extension and custom Language Server Protocol that catches RAM overruns, pin conflicts, strapping-pin boot failures, and bus configuration errors in Arduino and ESP32 code as you type, not after the flash. Built solo in a few hours at Hack Western 12, where it won Best Developer Tool.

Install from the VS Code Marketplace or Open VSX · 800+ combined installs.

1. Abstract

Embedded development still debugs like it is 1995. Write code, hit upload, watch the ESP32 reboot because a strapping pin was held low at reset, or because the RAM estimate was off by 30%. Web and native developers have type checkers, language servers, and instant diagnostics. Firmware engineers have a serial monitor and a multimeter. Sentinel is the bet that the same Language Server Protocol machinery that powers TypeScript and Rust tooling can model a microcontroller's hardware constraints, and surface them in the editor before the compile cycle.

The implementation is a custom Node.js Language Server paired with a VS Code extension, both written end-to-end during Hack Western 12. The server holds an incremental AST of every .ino and .cpp file in the project, plus a per-board hardware state machine: pin map, bus assignments, timer usage, interrupt vectors. A rule engine fires on every keystroke and reports diagnostics back through standard LSP messages. The extension renders them inline, plus an interactive board diagram that updates as pin assignments move around in source.

The novelty is the surface, not the analyses. Static analysis for embedded code is not new (PC-Lint, Clang-Tidy, Cppcheck all exist), but they all sit downstream of the compiler and outside the editor. Moving the same kind of analysis into the LSP layer makes the feedback loop the same one web developers have taken for granted for a decade: keystroke, diagnostic, fix, no flash needed.

2. Introduction

The compile / flash / reboot / debug cycle is the productivity tax embedded developers pay every day. The cost is not the compile time; the cost is everything that has to happen between writing a line of code and finding out that line was wrong. A typed misassignment on an ESP32 strapping pin is invisible until the chip refuses to boot. A buffer that overflows SRAM by 5% does not surface until the program crashes on a heap allocation an hour into operation. A wrong I2C clock speed against a slow sensor is a multi-hour scope session, not a compiler error.

  • The feedback loop is the bottleneck. Compiling and flashing an Arduino sketch is seconds; debugging why it crashed after deployment is hours. The closer diagnostics live to the keystroke, the faster the loop closes.
  • The hardware is the constraint. Most embedded bugs are not software bugs in the traditional sense. They are mismatches between code and silicon: pin role conflicts, bus contention, memory overflow, peripheral misconfiguration. A type checker that does not know about hardware is blind to them.
  • The tooling already exists for other languages. Web and systems developers have had LSP-grade diagnostics for years. The same protocol can carry hardware-aware diagnostics; nothing about LSP assumes the underlying analysis is purely textual.

2.1 Why the Language Server Protocol

LSP is the right substrate for two reasons. It already standardises the editor-to-analyzer wire format, so any editor that speaks LSP (VS Code, Neovim, Helix, Zed) gets Sentinel for free once the server is written. And it is built for incremental analysis: edits arrive as deltas, the server is expected to update diagnostics in milliseconds, and the editor handles the rendering. That posture matches what a hardware-aware analyzer needs to do anyway.

  • One server, many editors. The hardware analysis lives in a single Node.js process. Adding a new editor is a thin client, not a port.
  • Incremental by default. LSP semantics push the server toward keystroke-rate updates, which is exactly the latency target the build needs.
  • Diagnostic format is standardised. Severity, source, range, code, related information. The editor already knows how to render all of it. The server only has to fill it in.

2.2 What is actually new

The LSP framework, the VS Code extension surface, and the Arduino CLI build integration are all existing pieces. The contribution is the layer in between: a Node.js server that models board-specific hardware as first-class state, and that ships its analysis through the standard LSP diagnostic channel.

  • Hardware state as a first-class model. Pin assignments, bus configurations, interrupt vectors, strapping-pin behaviour at reset, and timer ownership are tracked per board across the whole project, not file by file.
  • Pre-flash hardware diagnostics. The pin conflicts, memory overruns, and strapping-pin hazards that normally surface as boot failures are surfaced as inline diagnostics during the edit, before any upload.
  • Editor-native, not toolchain-native. Sentinel does not replace avr-gcc or xtensa-esp32-elf-gcc. It lives upstream of both, in the editor, and surfaces what those compilers do not surface until link time or runtime.

Every section below is the engineering that makes those three commitments hold inside a Node.js LSP server built in a single hackathon session.

3. System

3.1 Memory budget estimation

Two budgets are tracked per board: SRAM and Flash. Both are computed statically from the AST, then compared against per-board ceilings (32 KB Flash and 2 KB SRAM on ATmega328P; 4 MB Flash and roughly 320 KB usable SRAM on ESP32 with WiFi active).

  • Static allocations from global variable declarations and array sizes. The cheap, accurate part of the estimate.
  • Stack estimated via per-function frame heuristics: local variable size plus a fixed call-depth scan from setup() and loop(). This is an approximation, not a worst-case bound. avr-gcc compresses frames in ways the source does not show, so the estimate is calibrated against compiled output rather than derived from first principles.
  • Framework overhead captured as per-board constants. Serial on ESP32 carries roughly 1 to 3 KB of RX/TX buffers and driver state. WiFi.begin() brings in tens to low hundreds of KB depending on what the stack ends up allocating at runtime; the live bar surfaces it as a step change so the cost is visible the moment the line is typed.
  • Compiler-specific calibration for avr-gcc and xtensa-esp32-elf-gcc, fitted against the compiled output of the calibration sketch suite across -O0, -O2, and -Os.
  • Live bars in the editor with thresholds at 80% (amber) and 90% (red). The number is an estimate. The colour is the actionable signal.
  • Ablation: removing the framework-overhead constants drops accuracy on the ESP32 calibration set from roughly 3% off compiled output to closer to 20%. Most of the work in memory estimation is in the constants, not the AST walk.

3.2 Pin conflict detection

Every GPIO assignment in the project is tracked across files and validated against the active board model. Pin role (input, output, INPUT_PULLUP, peripheral-claimed) is inferred from pinMode calls, bus initialisations, and analog APIs.

  • ESP32 strapping pins (GPIO0, GPIO2, GPIO5, GPIO12, GPIO15) are flagged with the specific boot-mode consequence: GPIO0 held low at reset enters download mode, GPIO2 must not be held high during flash, GPIO12 sets the flash voltage, GPIO15 controls the boot log on UART0.
  • Duplicate pin assignments across files. Two libraries claiming the same GPIO is a runtime failure the compiler does not see; Sentinel sees it because the hardware state machine is project-wide.
  • Hardware-only peripheral pins. SPI on the Uno must use pins 11, 12, 13. I2C on the Uno must use A4 and A5. A pinMode on any of those marks the pin as user-claimed even before SPI.begin() is reached.
  • INPUT_PULLUP detection for I2C lines. The analyzer cannot see external pull-up resistors on the board, but it can see the absence of an internal pull-up on a line the code uses for I2C, which is enough to surface a warning.
  • Interactive board diagram in the extension UI: pin assignments render with hover tooltips and click-to-source navigation. The diagram is the same data structure the rule engine reads from, rendered visually.

3.3 Bus configuration validation

  • I2C: Wire.setClock() against device-class defaults (standard 100 kHz, fast 400 kHz, fast-plus 1 MHz), 7-bit versus 10-bit addressing, and shared-bus address conflicts when multiple beginTransmission calls hit the same address without a multiplexer.
  • SPI: CPOL and CPHA match between master and device, MOSI / MISO / SCK pin assignments against the board's hardware SPI pins, and chip-select uniqueness across devices.
  • UART: baud rate, parity, stop bits, and which hardware UART is claimed. On the Uno, UART0 is shared with USB serial, so using it for a GPS module while also calling Serial.println for debug is a guaranteed conflict.
  • Peripheral autocomplete draws from a database of 200+ common Arduino-ecosystem sensors and breakouts (Adafruit, Sparkfun, generic datasheets). Address-based suggestion fires when an 0xXX literal is recognised as a known device's I2C address.
  • Ablation: dropping the device database collapses bus validation to syntax-only checks. The database is what turns "the bus is configured" into "the bus matches the device."

3.4 Language Server architecture

  • Incremental AST parser for .ino and .cpp, handling Arduino-specific preprocessing (auto-prototyping, header injection) and library resolution across sketch, user, core, and variant include paths.
  • Hardware state machine per board: pin map, bus assignments, timer ownership, interrupt vectors, peripheral states. The state machine is rebuilt incrementally from the AST on each edit rather than from scratch.
  • Validation rule engine spanning syntax, semantics, and hardware tiers. Rules are small pattern matches over the AST and the hardware state machine; new rules are a single function plus a registration call.
  • Performance: initial naive build hit 2 to 3 seconds per analysis. Incremental parsing, symbol-table caching, and rule batching cut typical edit latency to roughly 47 ms on the calibration sketch suite.
  • LSP diagnostic surface. Every rule's output is a standard LSP Diagnostic: severity, source sentinel, range pointing at the offending tokens, and a code that the extension uses to render the explanation panel.

4. Evaluation

Evaluation is small-scale and calibration-driven, in keeping with what a single developer can validate in a hackathon window. Numbers below are measured on the calibration sketch suite Sentinel ships with, not against an external benchmark.

  • Memory estimate accuracy: within roughly 3% of compiled-output RAM and Flash on the calibration sketches across the four supported boards. The estimate degrades on sketches that use heavy dynamic allocation (the String class on AVR, dynamic WiFiClient pools on ESP32) because those costs only materialise at runtime.
  • Diagnostic latency: 47 ms average from keystroke to inline diagnostic on sketches under 2000 lines. P95 under 100 ms on the same set.
  • Rule coverage: 50+ active rules across memory, GPIO assignment, strapping pins, and the three buses. New rules average tens of lines of code each, which is what makes the count plausible inside a hackathon window.
  • Adoption: 800+ combined installs across the VS Code Marketplace and Open VSX as a solo-maintained extension, post-event. The download path past the Hack Western audience is what makes the project something other than a demo.

These numbers are honest about the harness. The 3% memory estimate holds on the sketches the analyzer was calibrated against and degrades outside that distribution. The 50+ rule count is correct as a count, not as a claim that every rule is equally hardened: hackathon-grade rules and shipped-product-grade rules look the same in the rule engine, and a few are closer to the former. The 200+ device autocomplete database is mostly assembled from public datasheets rather than original research. Read the numbers as evidence of the editor-loop direction, not as a benchmark against PC-Lint or Clang-Tidy.

5. Conclusion and Implications

The state of embedded tooling is a productivity tax that limits who can build hardware. By moving diagnostics from after compile to inside the editor, Sentinel makes firmware development feel like high-level software development: fast iteration, instant safety rails, fewer trips to the oscilloscope. The safety-critical consequence is that the strapping-pin and pin-conflict failures that today surface as field-deployment bugs, bricked devices, or boot-loop incidents in industrial and medical embedded systems can surface as squiggles in the editor instead. Best Developer Tool at Hack Western 12 was the external validation that the gap is felt by the people who would benefit most from closing it; the 800+ installs since then are the quieter, more durable signal.

5.1 Limitations

  • Calibration scope. The 3% memory estimate is calibrated against a curated sketch suite. Sketches that lean on dynamic allocation drift wider, and the analyzer does not currently model heap fragmentation at all.
  • Stack estimation is approximate. Per-function frame heuristics cannot see the register allocation and frame-compression decisions avr-gcc and the Xtensa toolchain make, so the stack figure is a budgeted estimate, not a worst-case bound.
  • External pull-ups are invisible. Sentinel can see INPUT_PULLUP in code but cannot detect physical pull-up resistors on the board. I2C lines without an explicit pull-up in code get flagged conservatively, which produces false positives when the board supplies external pull-ups.
  • Board coverage is the original four. ATmega328P, ATmega2560, ESP32 (Xtensa LX6), ESP8266. The newer ESP32-S2 and S3 (Xtensa LX7) and the RISC-V ESP32-C3 / C6 are not modelled yet.
  • No FreeRTOS task-level analysis. Per-core stack tracking, task priorities, and FreeRTOS-style scheduling constraints are out of scope in the current release.
  • The 50+ rule count is uneven. Some rules are hardened against many sketches in the calibration suite; others are first-pass implementations that catch the obvious case and miss the edge.

5.2 Future Work

  • ESP32-S2 / S3 / C3 / C6 support. The Xtensa LX7 and RISC-V cores are where the ecosystem is moving; covering them is the single most useful expansion.
  • Per-region ESP32 memory breakdown: IRAM, DRAM, PSRAM, RTC. The current single-pool SRAM number hides the region that actually fills up first on real builds.
  • FreeRTOS task analysis. Per-task stack sizing, priority validation, and cross-core resource sharing on dual-core ESP32 builds.
  • Hardened rule audit. A second pass through the 50+ rules to bring the long tail up to the bar set by the strongest rules in the engine.
  • Timing analysis for I2C clock stretching, SPI setup and hold violations, and UART framing errors at high baud.
  • STM32 and RP2040. The natural next architectures to cover once the Xtensa LX7 work lands.
